# Observability Dashboard

Monitor agent-browser sessions in real time with a web dashboard showing a live browser viewport and command activity feed.

## Install

Download the dashboard once:

```bash
agent-browser dashboard install
```

This downloads the dashboard to `~/.agent-browser/dashboard/`. The files are served by the dashboard server once it is running.

## Usage

Start the dashboard server and open any session -- it appears automatically:

```bash
agent-browser dashboard start
agent-browser open example.com
```

Then open `http://localhost:4848` or your forwarded/proxied dashboard URL, such as the Portless URL `https://dashboard.agent-browser.localhost`, in a browser to see the live dashboard.

All sessions automatically stream to the dashboard. No extra flags are needed. The dashboard uses a same-origin proxy model, so the browser talks only to the dashboard origin while the server proxies session traffic internally to `127.0.0.1:<session-port>`. Per-session ports remain loopback-only implementation details.

### Session proxy routes

The dashboard server exposes these same-origin proxy routes:

- `GET /api/session/<port>/tabs` -> proxied to the session's `/api/tabs`
- `GET /api/session/<port>/status` -> proxied to the session's `/api/status`
- `WS /api/session/<port>/stream` -> proxied to the session's WebSocket stream

### Custom stream port

By default each session binds its WebSocket stream server to an OS-assigned port. To use a specific port, set the `AGENT_BROWSER_STREAM_PORT` environment variable:

```bash
AGENT_BROWSER_STREAM_PORT=9223 agent-browser open example.com
```

You can also use the runtime commands to control streaming on a running session:

```bash
agent-browser stream enable --port 9223
agent-browser stream status
agent-browser stream disable
```

## WebSocket protocol

The dashboard uses the same message types as [Streaming](/streaming), but it reaches session streams through the dashboard server's same-origin proxy routes:

### Command events

Sent when a command begins executing:

```json
{
  "type": "command",
  "action": "click",
  "id": "r123",
  "params": { "selector": "@e5" },
  "timestamp": 1711367000000
}
```

### Result events

Sent when a command finishes:

```json
{
  "type": "result",
  "id": "r123",
  "action": "click",
  "success": true,
  "data": {},
  "duration_ms": 45,
  "timestamp": 1711367000045
}
```

### Console events

Sent when the browser logs to the console:

```json
{
  "type": "console",
  "level": "log",
  "text": "Page loaded",
  "args": [{"type": "string", "value": "Page loaded"}],
  "timestamp": 1711367000100
}
```

The `args` array contains the raw CDP `Runtime.consoleAPICalled` arguments for programmatic access. Object arguments include preview data (e.g. `{userId: "abc", count: 42}` instead of `"Object"`).

These are in addition to the existing `frame`, `status`, and `error` message types documented on the [Streaming](/streaming) page.

## Architecture

The dashboard is a Next.js static export (`output: 'export'`) that produces plain HTML, CSS, and JS. It lives at `packages/dashboard/` in the monorepo and is built with:

```bash
pnpm build:dashboard
```

The built files are served by the dashboard server. The browser stays on the dashboard origin for both HTTP and WebSocket traffic, and the server proxies session-specific requests to `127.0.0.1:<session-port>` internally. This is why the dashboard works from local URLs and forwarded/proxied HTTPS workspace URLs without exposing per-session ports.

When the dashboard is not installed, visiting the HTTP endpoint shows instructions to run `agent-browser dashboard install`.
